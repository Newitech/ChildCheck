     1→     1→# ChildCheck — Secure Child Check-In / Check-Out System
     2→     2→
     3→     3→**A self-hosted, cross-platform, web-based child check-in / check-out system for churches (Seventh-Day Adventist focused), clubs, schools, and childcare organisations.**
     4→     4→
     5→     5→Inspired by the check-in/out components of Rock RMS and Elvanto, but re-imagined as a standalone, self-hosted, privacy-first system focused exclusively on the check-in / check-out workflow.
     6→     6→
     7→     7→---
     8→     8→
     9→     9→## 1. Vision & Goals
    10→    10→
    11→    11→### 1.1 What we are building
    12→    12→A single deployable application that lets an organisation (church, club, school, childcare centre, playgroup, Scouts troop, etc.) securely check children **in** to and **out** of programs, classes, and one-off events — while keeping **all personal and child data on the organisation's own infrastructure** (no cloud provider, no third-party data hosting).
    13→    13→
    14→    14→### 1.2 Primary target audience
    15→    15→- **Primary:** Seventh-Day Adventist churches running **Sabbath School**, **Pathfinders**, **Adventurers**, community childcare, and custom programs.
    16→    16→- **Secondary (via branding/config):** Any church (e.g. one that prefers the term "Sunday School"), Scouts, childcare centres, playgroups, schools, clubs.
    17→    17→
    18→    18→### 1.3 Core principles
    19→    19→1. **Self-hosted & local-first** — all data stays on the organisation's hardware/network. No mandatory cloud dependency.
    20→    20→2. **Cross-platform** — runs on Linux, Windows, macOS, NAS devices, and Docker.
    21→    21→3. **Web-accessible + installable** — a responsive web app that can be "installed" to the home screen / desktop (PWA) on Android, iOS, Windows, macOS, Linux.
    22→    22→4. **Secure by default** — personal + child data is encrypted at rest, access is role-based, all sensitive actions are audit-logged.
    23→    23→5. **Configurable, not custom-coded** — branding, terminology, and feature toggles live in an Admin Dashboard so the same binary serves an SDA church, a Scouts troop, or a childcare centre.
    24→    24→6. **Offline-resilient kiosk** — the kiosk keeps working during network blips and resyncs when connectivity returns.
    25→    25→
    26→    26→### 1.4 Non-goals (explicitly out of scope for this build)
    27→    27→- A full church management system (we are not rebuilding Rock/Elvanto — only the check-in/out slice).
    28→    28→- Cloud hosting / SaaS multi-tenancy. (Single organisation per install; multi-site is a *future* item.)
    29→    29→- Replacing the organisation's existing CRM/CHM. (Future import/export integrations are planned.)

    50→    50→| UC-14 | Guardian | Authorise an **older sibling** to collect a younger sibling. |

    54→    54→## 3. Technology Stack & Architecture
    55→    55→
    56→    56→### 3.1 Stack (locked)
    57→    57→| Layer | Choice | Rationale |
    58→    58→|-------|--------|-----------|
    59→    59→| Framework | **Next.js 16 (App Router) + TypeScript 5** | Already initialised; SSR + API routes in one binary; PWA-capable. |
    60→    60→| UI | **Tailwind CSS 4 + shadcn/ui (New York) + Lucide** | Consistent, accessible, already present. |
    61→    61→| State | **Zustand** (client) + **TanStack Query** (server) | Already available. |
    62→    62→| Auth | **NextAuth.js v4** (credentials provider, PIN/password, optional email recovery) | Already available; works fully offline. |
    63→    63→| ORM | **Prisma** | Already available. |
    64→    64→| DB (default) | **SQLite** (file-based, zero-config, perfect for self-hosting on a NAS) | Portable, ships inside the install. |
    65→    65→| DB (optional) | **PostgreSQL** (documented swap path in `.env`) | For larger orgs / multi-device write load. |
    66→    66→| Realtime | **Socket.io** mini-service (separate port, gateway-routed) | Live roster updates to teacher dashboard & kiosks. |
    67→    67→| Printing | **QZ Tray** or **node-thermal-printer** / browser print to configured queues | Cross-platform label printing. |
    68→    68→| Packaging | **Docker** (primary) + **Bun standalone binary** + **systemd/launchd/Winsw** service wrappers | Cross-platform deployment. |
    69→    69→
    70→    70→### 3.2 High-level architecture

    72→    72→┌──────────────────────────────────────────────────────────────┐

    77→    77→                            │ HTTPS (Caddy gateway, port 3000)
    78→    78→┌───────────────────────────▼──────────────────────────────────┐

    80→    80→│  ├─ NextAuth (PIN/password, session, RBAC)                    │
    81→    81→│  ├─ Domain APIs (people, families, check-in/out, reports)     │
    82→    82→│  └─ Audit log, backup/restore, import/export                  │
    83→    83→└──────────┬──────────────────────────────┬────────────────────┘
    84→    84→           │                              │
    85→    85→   ┌───────▼───────┐              ┌───────▼────────┐
    86→    86→   │ Prisma Client │              │ Socket.io svc  │
    87→    87→   │ SQLite/PG     │              │ (port 3003)    │
    88→    88→   └───────────────┘              └────────────────┘
    89→    89→           │
    90→    90→   ┌───────▼───────┐
    91→    91→   │ File storage  │  (photos, backups, label templates)
    92→    92→   │ ./data/       │  encrypted at rest via app-layer AES
    93→    93→   └───────────────┘
    94→    94→```
    95→    95→
    96→    96→### 3.3 Cross-platform packaging strategy
    97→    97→- **Docker** — primary, recommended for NAS / Linux servers. Single `docker-compose.yml` with volumes for `./data` and `./db`.
    98→    98→- **Bun standalone binary** — `bun build --compile` produces a single executable for Linux/macOS/Windows. Bundles the Next.js standalone output.
    99→    99→- **Service wrappers** — systemd unit (Linux), launchd plist (macOS), winsw/nssm (Windows), Synology DSM task (NAS).
   100→   100→- **Install scripts** — one-shot bash (Linux/Mac) and PowerShell (Windows) scripts in `/install/`.
   101→   101→
   102→   102→### 3.4 PWA / "install like an app"
   103→   103→- Web manifest with `display: standalone`, maskable icons, theme colour pulled from branding config.
   104→   104→- Service worker for offline shell + cached kiosk assets.
   105→   105→- On iOS/Android/Chrome/Edge, "Add to Home Screen" / "Install" produces an app-like icon with no browser chrome.
   106→   106→- Same binary serves kiosk tablets, guardian phones, admin desktops.

   110→   110→## 4. Core Domain Model (high level)
   111→   111→
   112→   112→> Full Prisma schema is defined stage-by-stage. This is the conceptual map.
   113→   113→
   114→   114→### 4.1 Organisation & Configuration
   115→   115→- **Organisation** (singleton) — name, branding (logo, colours, terminology overrides: e.g. "Sabbath School" vs "Sunday School" vs "Unit").
   116→   116→- **FeatureFlags** — every enable/disable toggle in the Admin Dashboard (see §6).
   117→   117→- **Printer** — name, driver type, assigned area/classroom.
   118→   118→- **Area / Room** — physical locations where check-in/out happens.
   119→   119→
   120→   120→### 4.2 Programs, Classes, Events
   121→   121→- **Program** — recurring program (Sabbath School, Pathfinders, Adventurers, Community Childcare, Custom…).
   122→   122→- **Group / Class** — a class within a program (e.g. "Beginner", "Kindergarten", "Primary", "Juniors"). Assigned to a Room and a Schedule.
   123→   123→- **Schedule** — when a class meets (day-of-week + time, or ad-hoc).
   124→   124→- **Event** — a one-off / occasional event not under a regular program (e.g. "Community Fun Day 2025"). Can reuse classes or define ad-hoc rooms.
   125→   125→
   126→   126→### 4.3 People & Families
   127→   127→- **Person** — a human. Has a `personType`: `Adult` | `Child`. Common fields + type-specific fields.
   128→   128→  - Adult-only fields: email, phone, PIN, password hash, self-manage flag.
   129→   129→  - Child-only fields: DOB, school grade, allergies, medical notes, photo-verification flag.
   130→   130→- **Family** — a household. Has 1+ **PrimaryCarer** (Adult Person) and 0+ Child Persons.
   131→   131→- **FamilyMembership** — join table Person↔Family with role (`PrimaryCarer`, `Child`, `AuthorisedGuardian`, `EmergencyContact`).
   132→   132→- **AuthorisedGuardian** — an Adult Person (may belong to their *own* Family) who can sign in/out a given Family's children but cannot edit that family's data.
   133→   133→- **BlacklistEntry** — Person (or free-text name+description) explicitly **not** permitted to collect a specified child/family.
   134→   134→- **OlderSiblingAuthorisation** — flag + details allowing an older sibling Person to collect a younger sibling.
   135→   135→- **WorkingWithChildrenCard** — card type (e.g. "QLD Blue Card", "NSW WWCC", international equivalent), number, status, expiry, verified-by.
   136→   136→
   137→   137→### 4.4 Users & Security Roles
   138→   138→- **User** — a login account. **Linked 1:1 to a Person** (Admin, Teacher, Volunteer, Kiosk all link from PEOPLE records and share the Person's credentials).
   139→   139→- **Role** — `Admin`, `Security`, `Teacher`, `Volunteer`, `Kiosk`, `PeopleManager`.
   140→   140→- **RoleAssignment** — User↔Role with optional scope (e.g. Teacher scoped to a Class).
   141→   141→- **Permission** — granular actions (`check_in`, `check_out`, `override_checkout`, `view_roster`, `manage_people`, `manage_programs`, `manage_system`, `run_reports`, `backup_restore`).
   142→   142→
   143→   143→### 4.5 Check-In / Check-Out
   144→   144→- **Session** — a check-in "day" for a Program/Class/Event (e.g. "Sabbath School — 2025-01-18").
   145→   145→- **CheckInRecord** — Child Person, Session, Class/Room, checked-in-by (User/Guardian), timestamp, generated **DailyCode** (3-digit), label-printed flag.
   146→   146→- **CheckOutRecord** — Child Person, Session, checked-out-by, timestamp, method (`Code` | `PIN` | `Override`), override note (if any), photo-verified flag.
   147→   147→- **DailyCode** — a random 3-digit code generated per **family per day** (visible to all carers/guardians of that family who sign in with PIN). Used for quick sign-out.
   148→   148→- **HeadcountLog** — Teacher-entered headcount snapshots per Room/Session.
   149→   149→- **OverrideCheckoutLog** — mandatory confirmation checkbox + free-text note + authorising User, for every override.
   150→   150→
   151→   151→### 4.6 Audit
   152→   152→- **AuditLog** — actor, action, entity, before/after, timestamp, IP. Append-only, tamper-evident (chained hash).
   153→   153→

   156→   156→## 5. Security Model
   157→   157→
   158→   158→| Concern | Approach |
   159→   159→|---------|----------|
   160→   160→| Data residency | 100% on the organisation's hardware. No outbound data calls except optional SMTP for email recovery/notifications. |
   161→   161→| Encryption at rest | Photos + backups encrypted with AES-256-GCM (key derived from a master secret in env/config, rotatable). DB file permissions locked to the service user. |
   162→   162→| Passwords | bcrypt-hashed. Optional PIN (4–6 digits) for kiosk/guardian quick login. |
   163→   163→| Sessions | HTTP-only, Secure, SameSite=Strict cookies; configurable session TTL; idle timeout on kiosk. |
   164→   164→| RBAC | Role → Permission matrix enforced in API route handlers (server-side, never trust client). |
   165→   165→| Audit | Every check-in/out, override, data change, config change, login written to append-only audit log. |
   166→   166→| Rate limiting | Login + kiosk search rate-limited to prevent enumeration. |
   167→   167→| Backup | Encrypted, downloadable, restorable. Scheduled option. |
   168→   168→| Network | Caddy terminates TLS; recommend LAN-only exposure by default. Documented hardening guide. |
   169→   169→| Child data minimisation | Photos optional + toggleable; medical/allergy fields shown only to authorised roles; visitor records purgeable. |

   173→   173→## 6. Feature Toggles (Admin Dashboard)
   174→   174→
   175→   175→Every toggle below is an **enable/disable** switch in the Admin Dashboard, stored in `FeatureFlags`.
   176→   176→
   177→   177→| Toggle | Default | Effect when ON |
   178→   178→|--------|---------|----------------|
   179→   179→| `kiosk_requires_login` | OFF | Kiosk requires a Kiosk-Account PIN before showing search. |
   180→   180→| `guardian_pin_signin` | ON | Guardians/Authorised Guardians can sign in with PIN/password at kiosk to sign children in/out. |
   181→   181→| `guardian_self_registration` | OFF | Guardians can self-register & self-manage family data + PIN/password. |
   182→   182→| `email_recovery` | OFF | Allow email-based password recovery (requires internet/SMTP). |
   183→   183→| `email_as_contact` | ON | Email stored as a contact/communication method (not used for auth). |
   184→   184→| `photo_verification` | ON | Photos required/used for child + adult verification at checkout. |
   185→   185→| `print_name_labels` | ON | Print a child name label at check-in. |
   186→   186→| `print_signout_code` | ON | Print the daily 3-digit code + details for the guardian at check-in. |
   187→   187→| `override_checkout` | OFF | Admins/Teachers may override checkout (with confirmation + note). |
   188→   188→| `older_sibling_collect` | OFF | Allow authorising an older sibling to collect a younger sibling. |
   189→   189→| `visitors_add_to_db` | ON (prompt) | Visitor flow offers "add to regular database" checkbox. |
   190→   190→| `working_with_children_tracking` | ON | Track WWCC / Blue Card status for volunteers. |
   191→   191→| `audit_log_detailed` | ON | Verbose audit logging. |
   192→   192→| `scheduled_backups` | OFF | Automated encrypted backups on a schedule. |

   257→   257→
   258→   258→### PHASE C — The Kiosk & Check-In/Out (the heart of the system)
   259→   259→
   260→   260→#### Stage 6 — Kiosk shell & search
   261→   261→- Kiosk route group `(kiosk)` with fullscreen, touch-friendly UI.
   262→   262→- Open mode vs locked mode (per `kiosk_requires_login`).
   263→   263→- Universal search across name, surname, phone, email (rate-limited, no partial-data leakage).
   264→   264→- Family selection → family summary card (carers, children, alerts).
   265→   265→- Idle reset back to search after configurable timeout.
   266→   266→- **Deliverable:** a tablet can be set up as a kiosk and a guardian can find their family.
   267→   267→
   268→   268→#### Stage 7 — Check-In flow
   269→   269→- Select Program/Class/Event + Session (today's session auto-suggested).
   270→   270→- Multi-child selection with checkboxes/sliders (no dropdown-per-child).
   271→   271→- Allergy/medical alerts surfaced prominently per child.
   272→   272→- Authorised-guardian flow: guardian signs in with PIN/password (if `guardian_pin_signin` ON) → can act for the family.
   273→   273→- Generate **DailyCode** (3-digit, per family per day) — shown on screen, visible to all carers/guardians of that family who sign in.
   274→   274→- Optional label printing (`print_name_labels`) + optional signout-code slip printing (`print_signout_code`).
   275→   275→- Visitor quick-add flow with "add to regular DB" checkbox (per `visitors_add_to_db`).
   276→   276→- Write CheckInRecord + AuditLog.
   277→   277→- **Deliverable:** a guardian can check in 3 children across 2 rooms in under 20 seconds and walk away with a code + labels.
   278→   278→
   279→   279→#### Stage 8 — Check-Out flow
   280→   280→- Three checkout methods:
   281→   281→  1. **Code** — enter the family's daily 3-digit code → quick sign-out of some/all children (checkboxes).
   282→   282→  2. **PIN** — guardian PIN/password → sign-out.
   283→   283→  3. **Override** — Admin/Teacher only (per `override_checkout`): mandatory confirmation checkbox + free-text note → OverrideCheckoutLog.
   284→   284→- Photo verification display (per `photo_verification`): show child photo + collector photo for visual match.
   285→   285→- Blacklist hard-stop: if the collector matches a BlacklistEntry, block + alert Security.
   286→   286→- Older-sibling collection (per `older_sibling_collect`): allow if authorised.
   287→   287→- Write CheckOutRecord + AuditLog.
   288→   288→- **Deliverable:** the daily code works for fast checkout; override is audited; blacklist blocks.
   289→   289→

   432→   432→## 9. Development Conventions
   433→   433→
   434→   434→- **Frontend first per stage:** build the UI so the user can see progress, then wire the API routes + Prisma.
   435→   435→- **API routes** (not server actions) for all data mutations.
   436→   436→- **Prisma schema** evolves stage-by-stage; `bun run db:push` after each schema change.
   437→   437→- **shadcn/ui** for all UI primitives; domain components in `src/components/domain/`.
   438→   438→- **Sticky footer** on every layout (`min-h-screen flex flex-col`, `mt-auto` footer).
   439→   439→- **Responsive + accessible** (ARIA, keyboard, 44px touch targets) from day one.
   440→   440→- **Self-verification:** after each stage, use Agent Browser to confirm the page renders and the golden path works before declaring done.
